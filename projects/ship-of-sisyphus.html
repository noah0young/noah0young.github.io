Ship of Sisyphus

About

Ship of Sisyphus is a game where you try to build a machine to climb a mountain. The game is broken up into two phases: the Build phase and Climb Phase. In the Build phase, you spend money to buy components to construct a climbing machine. Once you've finished, you go to the Climb Phase, where you watch your machine try to climb the mountain. At some point in this phase, your machine will get stuck or run out of battery. At this point, the phase will end and you will earn money for however high your machine was able to climb. Afterwards, you will be sent back to the Build Phase to repeat the addicting and fun cycle.

What I Did

For this project, my team divided up the work into three parts: the build phase, the climb phase, and the components of the vehicle. I worked on the climbing phase of the game. This meant I worked on the generation of the level and the checks for if the phase should continue.

The procedural generation system consisted of a ClimbManager, and ProceduralGenerator, . The ClimbManager placed the chunks of level and reacted to the player reaching the end of each level chunk to place the next chunk. To hide this process from the user, the game always begins by placing 7 level chunks and waits to delete old ones and place new ones until the player reaches the end of the third level chunk. This is performed by using a callback function for when the user reaches the end of a level chunk to add a new chunk. This algorithm stops placing new chunks once the final level chunk is placed, which allows the user to reach the end of the game. As a result, there are always at most 7 level chunks active in the game at a time.

The ProceduralGenerator was an object-oriented approach to determining which level chunks should be placed for each given area. This system worked by having the ClimbManager hold as many ProceduralGenerator objects as there were areas in the game and switching ProceduralGenerator after a set number of level chunks were completed. Every ProceduralGenerator referred to its data stored in its ProceduralData, which determined what level chunks got placed, the random weights given to each chunk, and when this ProceduralGenerator would end. The combination of these scripts made the logic of the climbing phase.

The ClimbingMachineManager handled the logic that determined if the climb phase should continue. This employed a heuristic algorithm that verified that the climbing machine was climbing the mountain at a predetermined rate. If it ever failed to meet that rate, it would report that to the ClimbManager which ended the climb phase.
